# 🧠 深度需求與架構對齊技能 (`grill-with-docs`)

此技能強制要求 AI 助理在動筆編寫任何程式碼之前，先與開發者進行「深度對齊」。拒絕憑感覺編碼，建立一致的技術語言與設計共識。

## 📋 核心流程說明

當此技能被觸發時，AI 助理**必須**遵循以下嚴格步驟：

### 步驟 1：暫停程式碼撰寫
* **絕不**在尚未釐清需求前修改或新增任何程式程式碼檔案。
* 告知使用者：「我已啟用 `/grill-with-docs` 技能。在開始撰寫程式碼之前，我需要與您進行深度需求對齊。」

### 步驟 2：進行結構化訪談
AI 助理必須向開發者提出 **3 到 5 個關鍵問題**，涵蓋以下面向：
1. **業務目標**：這個功能的最終商業目的與使用者場景是什麼？
2. **架構與依賴**：會影響到哪些既有模組？是否有特定的第三方 API、資料庫結構或環境限制？
3. **邊界情況 (Edge Cases)**：有哪些特殊的輸入值、網路延遲或例外狀況需要處理？
4. **定義成功**：我們如何驗證這個功能已完美實作？（例如：是否有測試標準或特定的輸出格式？）

> [!IMPORTANT]
> 每次提問應聚焦且精準，避免一次丟出過多問題讓開發者感到疲憊。

### 步驟 3：建立或更新 `CONTEXT.md`
訪談結束後，AI 助理必須在專案根目錄建立或更新 [CONTEXT.md](file:///c:/Users/chang/OneDrive/Documents/AG/CONTEXT.md)，其結構應包含：
* **通用語言 (Ubiquitous Language)**：定義本專案關鍵的名詞術語，避免認知分歧。
* **技術堆疊與限制**：明確記載使用的框架版本、程式語言特性、雲端服務（如 GCP Firestore, Cloud Run 等）。
* **目前架構設計**：用簡要的 Mermaid 圖表或文字描述模組之間的調用關係。
* **實作進度表**：清晰列出 `[ ] 待辦`、`[/] 進行中`、`[x] 已完成` 的任務清單。

### 步驟 4：開發者簽核 (Sign-off)
* 展示建好的 `CONTEXT.md` 內容。
* 詢問開發者：「這是我們對齊後的專案上下文與執行計畫，請問是否有需要修改的地方？若無誤，請回覆『確認』，我將正式開始實作第一步。」
* **在開發者回覆「確認」或表示同意之前，絕對不開始撰寫程式碼**。
