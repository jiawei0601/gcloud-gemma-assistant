# 🍰 垂直切片開發技能 (`vertical-slice`)

在面對大型、複雜的功能開發（如 Epic）時，傳統的「水平層次實作」（先寫完所有的資料庫表，再寫完所有的後端 API，最後才開始寫前端 UI）會帶來極高的集成風險，且中途無法交付任何有價值的成果。

此技能引導 AI 助理採用**「垂直切片（Vertical Slicing）」**，每次只做一個跨越所有技術層級的最小可行功能切片。

---

## 📐 垂直切片 vs 水平分層

```text
[ 水平分層 - ❌ 不推薦 ]
┌─────────────────────────────┐  <- 前端介面 (最後做，完全無法提早測試)
├─────────────────────────────┤  <- 業務邏輯 API (中間做)
└─────────────────────────────┘  <- 資料庫/存儲層 (最先做，盲目設計)

[ 垂直切片 - 🟢 推薦 ]
  切片 1      切片 2      切片 3
 ┌──────┐    ┌──────┐    ┌──────┐
 │ 前端 │    │ 前端 │    │ 前端 │   每個切片都是一個「完整、可運作」
 ├──────┤    ├──────┤    ├──────┤   的端到端（End-to-End）功能。
 │ 後端 │    │ 後端 │    │ 後端 │   可以提早交付、測試並獲得反饋。
 ├──────┤    ├──────┤    ├──────┤
 │ 數據 │    │ 數據 │    │ 數據 │
 └──────┘    └──────┘    └──────┘
```

---

## 📋 垂直切片實作步驟

當啟用 `/vertical-slice` 技能時，AI 助理**必須**遵循以下開發步驟：

### 步驟 1：定義與拆解切片
1. **盤點大功能**：將整個大型任務列出來。
2. **拆解為垂直切片**：以「使用者故事」或「獨立行為」為中心進行拆解。
   * *範例*：對於一個「任務管理系統」，不要拆解成「實作 PostgreSQL 連線、寫 REST API、刻 React 畫面」。
   * *應拆解為*：
     * **切片 1**：使用者可以在畫面上「新增任務」並儲存到資料庫（只做這個行為的 End-to-End）。
     * **切片 2**：使用者可以「讀取並顯示」任務列表。
     * **切片 3**：使用者可以「編輯與刪除」既有任務。

### 步驟 2：排定優先順序
* 找出**最具技術風險**或**核心價值最高**的切片作為「切片 1」（通常是涉及外部系統整合或核心演算法的切片）。
* 與開發者確認切片優先順序：「我們將首先攻克『切片 1』，因為它能最早驗證我們的技術可行性。」

### 步驟 3：迭代開發切片 (小步交付)
針對每一個垂直切片：
1. **精準設計**：只設計該切片所需的最小資料庫欄位、API 路由與前端元件，絕不過度設計（YAGNI - You Aren't Gonna Need It）。
2. **端到端實作**：
   * 寫資料庫 Migration / schema 變更。
   * 實作後端控制器與業務邏輯。
   * 實作前端互動介面與串接。
3. **完成測試與驗證**：運行單體測試與集成測試，確認該切片功能完美無瑕。
4. **Git Commit**：將這個完整的垂直切片進行 Commit。

### 步驟 4：收集反饋並調整
* 將運作良好的切片成果展示給開發者。
* 依據反饋微調下一個切片的計畫，保持敏捷度。
