# 🔍 結構化 Bug 診斷與修復技能 (`diagnose`)

當系統出現錯誤、異常或非預期行為時，AI 助理往往會急於猜測問題並直接修改代碼，這常導致「修好一個 Bug，引入三個 Bug」。此技能強制執行科學的診斷與修復流程。

## 🛠️ 嚴格診斷工作流 (Scientific Diagnosis Workflow)

當啟用 `/diagnose` 技能時，AI 助理**必須**遵循以下四個步驟：

### 📋 步驟 1：收集錯誤特徵與上下文 (Observe)
* **不要急於修改！** 先調用工具讀取相關的錯誤日誌（Stderr）、系統日誌、環境變數以及出錯的原始碼檔案。
* 整理並向開發者總結：
  * 出錯的具體位置（檔案路徑、行號、調用棧 Call Stack）。
  * 觸發錯誤的輸入參數或外部狀態。
  * 錯誤的類型與訊息（例如：`NullPointerException`, `KeyError`）。

### 🧪 步驟 2：建立重現測試 (Reproduce)
* **核心原則**：無法重現的 Bug 就無法保證被修復。
* 撰寫一個輕量級的重現指令碼（Scratch Script）或單體測試案例，以最乾淨的環境重現該問題。
* 執行該重現步驟，並將**「失敗的測試輸出/錯誤日誌」**截圖或以純文字呈現給開發者，證明我們已成功在受控環境下 100% 重現問題。

> [!NOTE]
> 重現指令碼應放置在臨時資料夾（如專案的 `scratch/` 或 `tests/`）中，並命名為 `reproduce_issue_xxx.py` 等。

### 🧠 步驟 3：根因分析與方案設計 (Analyze & Design)
* 撰寫一份簡短的 **根因分析 (RCA, Root Cause Analysis)** 說明：
  * **表面現象**：系統在何處崩潰或輸出錯誤。
  * **深層原因**：是什麼樣的邏輯漏洞、競爭條件（Race Condition）、或狀態不一致導致了這個問題。
* 提出 **2 到 3 種修復方案**，並分析各方案的優缺點（Pros & Cons），例如：
  * *方案 A*：快速修補（Quick Fix），改動小但可能留下技術債。
  * *方案 B*：結構化重構（Refactoring），代碼更乾淨但影響層面較廣。
* **等待開發者評估並選擇方案**。

### 🚀 步驟 4：精準修復與雙重驗證 (Verify)
* 在獲得開發者授權的修復方案後，精準修改對應的程式碼檔案。
* **執行雙重驗證**：
  1. **驗證重現步驟**：運行步驟 2 建立的重現測試，確認該測試已順利通過（🟢）。
  2. **驗證迴歸測試**：運行專案中既有的完整測試套件，確保此次修復沒有破壞系統的其他功能。
* 驗證通過後，將重現指令碼或測試檔案整理（如果不再需要則刪除，若有價值則納入正式測試庫），並執行 Git Commit。
